# logi-cli V1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a macOS-first `logi-cli` V1 for Logitech MX Master 4 that provides a `CLI + daemon` runtime, TOML configuration, per-app bindings, gesture handling, and diagnostics.

**Architecture:** Ship a single Go binary with a narrow runtime pipeline: HID report input, event normalization, rule matching, and action execution. Keep device-specific decoding isolated from the rule engine, and keep macOS-only behavior behind platform adapters so the daemon core stays testable without hardware.

**Tech Stack:** Go, `github.com/sstallion/go-hid`, `github.com/spf13/cobra`, `github.com/BurntSushi/toml`, macOS CoreGraphics/AppKit bindings via cgo, `launchctl`, Go's standard `testing` package.

---

## Scope Check

The approved spec is one cohesive V1, not multiple unrelated subprojects. The tasks below keep the implementation sequence narrow: first bootstrap the binary and config model, then make hardware observable, then layer in device decoding, rules, macOS actions, and daemon control.

Use `@systematic-debugging` if HID decoding or macOS permission behavior differs from expectations. Use `@verification-before-completion` before claiming MX Master 4 support works on hardware.

		actionIDs[action.ID] = struct{}{}
	}
	for _, profile := range cfg.Profiles {
		for _, binding := range profile.Bindings {

			}
		}
	}
	return nil
}

			MfrStr:     info.MfrStr,

	})

	switch action.Type {
	case "shortcut":
		return e.ShortcutEmitter.Emit(ctx, action.Keys)
	case "system":
		return e.SystemRunner.Run(ctx, action.System)
	case "script":
		return e.ScriptRunner.Run(ctx, action.Script)
	default:
		return fmt.Errorf("unsupported action type %q", action.Type)
	}
}
